ğŸ“¦ 2. Modules

What modules are and how they organize content features

What is a Module?

A module is a category or section of the website that organizes a specific type of content. Think of modules as different "departments" of the website, each dedicated to a particular kind of information.

Each module has its own page on the frontend where visitors can browse all entries within that module.

How Modules Work

Every module has:

  • Name: What the module is called (e.g., "Sacred Stories")
  • Slug: The URL-friendly name (auto-generated from the name)
  • Icon: An emoji that represents the module visually
  • Entries: Individual pieces of content within the module

ğŸ“‹ 4. JSON (JavaScript Object Notation)

What JSON is and how it's used in the system

What is JSON?

JSON is a simple way to store structured data as text. Think of it like a very organized list. The system uses JSON internally to store things like custom field values, field visibility settings, and more.

You don't need to write JSON manually! The admin panel handles it for you. But understanding what it looks like helps if you ever need to troubleshoot.

What JSON Looks Like

{
  "opening_time": "5:00 AM",
  "closing_time": "9:00 PM",
  "entry_fee": 0,
  "has_darshan": true,
  "importance_level": 5
}

How JSON is Used Here

  • Field Visibility: When you toggle which fields are visible on the frontend, the on/off state is saved as JSON
  • Custom Field Values: When you fill in custom fields (like opening time, entry fee), these are stored as JSON
  • Gallery Image Paths: Multiple image paths are stored as comma-separated text (similar concept)
ğŸ“Œ Real Example: Field Visibility JSON
{"title": 1, "description": 1, "featured_image": 1, "latitude": 0}

Here, 1 means "show on frontend" and 0 means "hide". Title, description, and image are shown; latitude is hidden.

ğŸ’¡ Key takeaway: You never need to write JSON yourself. The admin forms handle everything. JSON is just how the system stores your data behind the scenes.

ğŸ“¸ 6. Images & Gallery

How to manage images across all tiers

Image Types

  • Featured Image: The main image for any tier. This appears as the primary image on the frontend page and in cards/listings. Each item has ONE featured image.
  • Gallery Images: Additional images shown in a slider on the frontend. Items can have MULTIPLE gallery images. Available for T1 (via Gallery upload), T2, T3, and T4.

How Images Work

When you upload an image:

  • It's saved to the server's static/uploads/images/ folder
  • A unique filename is generated to prevent conflicts
  • A thumbnail appears in the form so you can see what's uploaded
  • Click any thumbnail to see the full-size image (lightbox)
  • Click the red âœ• on a thumbnail to delete it (with double confirmation)

Image Best Practices

  • Featured Image: Use a high-quality landscape photo (recommended: 1200Ã—800 or larger)
  • Gallery Images: Upload 3-10 images showing different aspects of the place
  • File formats: JPG, PNG, GIF, and WebP are supported
  • File size: Keep images under 5MB for best performance
  • Naming: The system auto-renames files, so original names don't matter

Hero Slider on Homepage

The homepage hero slider automatically pulls images from ALL tiers that have uploaded images. It picks featured images and gallery images randomly and shows them in an auto-rotating slideshow with tier labels and "View" buttons.

ğŸ’¡ Tip: Upload at least one image per Tier 1 dham to ensure the homepage looks beautiful. The more images across all tiers, the more diverse the hero slider becomes!

ğŸ’¡ 7. Tips & Best Practices

Helpful advice for managing content effectively

Content Creation Order

Always create content top-down:

  • First create a T1 Holy Dham (e.g., Vrindavan)
  • Then add T2 Key Places within it (e.g., Vrindavan Town, Barsana)
  • Then add T3 Key Spots within each Key Place (e.g., Banke Bihari Temple)
  • Finally add T4 Key Points within each Key Spot (e.g., Jhulan Sthal)

Publishing Workflow

  • Draft: Content is saved but not visible to visitors
  • Published: Content is live and visible on the frontend
  • Start with Draft to prepare content, then switch to Published when ready

Search & Discovery

The frontend has a live search that searches across ALL 4 tiers simultaneously. Make sure to use descriptive titles and add location data (city, state) so visitors can find places easily.

Deleting Content

âš ï¸ Important: All delete operations require TWO confirmations (double confirmation). The first asks "Are you sure?" and the second asks "This is irreversible. Are you absolutely sure?" This is to prevent accidental deletion.
